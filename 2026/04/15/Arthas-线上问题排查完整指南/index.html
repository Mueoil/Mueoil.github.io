Arthas 简介

Arthas 是 Alibaba 开源的 Java 诊断工具，用于线上问题排查。它可以在不重启服务的情况下，实现对 JVM 状态的监控、方法的追踪、类的热更新等操作。

核心特性

  • 无侵入：无需修改应用代码，直接 attach 到目标 JVM
  • 功能全面：涵盖 JVM 监控、方法追踪、类加载、线程分析等
  • 在线热修复：支持在线修改类文件
  • 命令交互友好：提供类似 shell 的交互体验

快速开始

1. 安装与启动

# 下载 arthas-boot.jar
curl -O https://arthas.aliyun.com/arthas-boot.jar

# 运行
java -jar arthas-boot.jar

# 选择目标进程，输入进程编号即可 attach

2. 基本操作

# 退出会话
quit

# 完全退出（关闭 arthas-server）
stop

# 查看帮助
help

常用排查场景

CPU 高占用排查

场景描述

系统 CPU 使用率持续偏高，导致服务响应变慢或超时。

排查步骤

步骤1：定位高 CPU 占用线程

# 查看当前所有线程的 CPU 使用情况
thread

# 查看 CPU 使用率最高的前 10 个线程
thread -n 10

# 查看指定线程的堆栈信息
thread <thread-id>

步骤2：分析线程堆栈

# 查看线程详情（含堆栈）
thread <thread-id>

# 查看所有线程的堆栈
thread -a

# 筛选特定状态的线程
thread --state BLOCKED
thread --state WAITING
thread --state TIMED_WAITING

步骤3：监控 CPU 变化

# 开启 CPU 监控，每 1000ms 输出一次，共 5 次
thread -n 10 -i 1000 -c 5

步骤4：定位热点方法

# 开启方法执行时间统计
profiler start

# 查看当前统计结果
profiler getSamples

# 停止并生成报告
profiler stop

# 生成火焰图
profiler stop --format svg --file cpu_flamegraph.svg

完整示例

# 1. 查看 CPU 最高的线程
thread -n 5

# 假设输出显示线程 ID 1234 CPU 占用最高
# 2. 查看该线程堆栈
thread 1234

# 3. 如果堆栈不清晰，使用 profiler
profiler start --event cpu
# 等待一段时间
profiler stop

内存泄漏排查

场景描述

应用内存持续增长，最终导致 OutOfMemoryError 或频繁 Full GC。

排查步骤

步骤1：查看内存概况

# 查看 JVM 内存信息
memory

# 查看堆内存详情
vmtool --action getInstances --className java.lang.String --limit 10

步骤2：查找大对象

# 查找堆中所有 String 对象
vmtool --action getInstances --className java.lang.String --limit 100

# 查找特定类的实例数量
sc -d *ClassName*

# 查找占用内存大的对象
vmtool --action getInstances --className java.util.HashMap --limit 20 --displayFieldValues

步骤3：分析类加载器

# 查看所有已加载的类
classloader

# 查看特定类加载器加载的类
classloader -l

# 查看类的加载详情
classloader -c <classloader-hash> -r

步骤4：触发 GC 并观察

# 手动触发 GC
vmtool --action getInstances --className java.lang.Object --limit 1 --expr =@Class@forName("java.lang.System").getMethod("gc", null).invoke(null, null)

# 或者
# jcmd <pid> GC.run

步骤5：dump 堆内存

# 生成堆 dump
heapdump /tmp/heapdump.hprof

# 离线分析 dump 文件
jhat -J-Xmx4G /tmp/heapdump.hprof
# 或使用 MAT、VisualVM 等工具分析

完整示例

# 1. 查看内存使用情况
memory

# 2. 查找疑似泄漏的对象（如自定义缓存类）
vmtool --action getInstances --className com.xxx.cache.UserCache --limit 100

# 3. 查看对象详情
vmtool --action getInstances --className com.xxx.cache.UserCache --limit 1 --displayFieldValues

# 4. 如果确认泄漏，dump 堆分析
heapdump /tmp/heapdump.hprof

接口响应慢排查

场景描述

某接口响应时间变慢，需要定位具体是哪个方法或数据库查询导致。

排查步骤

步骤1：追踪方法调用链

# 追踪方法调用栈
trace com.xxx.service.UserService getUserInfo

# 只追踪耗时超过 100ms 的调用
trace com.xxx.service.UserService getUserInfo '#cost > 100'

# 追踪某个包下的所有方法
trace com.xxx.service.*

# 排除某些包
trace com.xxx.service.* '#cost > 100 && !@java.lang.Thread@currentThread().getName().startsWith("Arthas")'

步骤2：监控方法执行次数和耗时

# 监控方法调用次数和平均耗时
monitor -c 5 com.xxx.service.UserService getUserInfo

# 监控某个包下的方法
monitor com.xxx.controller.*

# 指定监控条件
monitor -c 5 com.xxx.service.UserService getUserInfo '#cost > 500'

步骤3：查看方法内部调用

# 深入查看方法内部执行情况
stack com.xxx.service.UserService getUserInfo

# 条件过滤
stack com.xxx.service.UserService getUserInfo '#cost > 200'

步骤4：方法计时统计

# 统计方法执行时间分布
tt -t com.xxx.service.UserService getUserInfo

# 查看 tt 记录
tt -l

# 查看特定记录详情
tt -i <index>

# 重放调用
tt -i <index> -p

完整示例

# 1. 粗略定位：查看哪个 controller 方法慢
trace com.xxx.controller.* *

# 2. 定位到具体 service 方法后，深入追踪
trace com.xxx.service.OrderService createOrder

# 3. 如果还看不清，用 tt 捕获调用
tt -t com.xxx.service.OrderService createOrder '#cost > 1000'

# 4. 调用几次后，查看捕获的记录
tt -l

# 5. 查看具体记录
tt -i 1001

# 6. 查看方法内部调用栈
stack com.xxx.dao.OrderDao insert

线程阻塞/死锁排查

场景描述

系统出现大量线程阻塞，或疑似死锁导致请求无法响应。

排查步骤

步骤1：查看线程状态分布

# 查看所有线程状态
thread

# 查看 BLOCKED 状态线程
thread --state BLOCKED

# 查看 WAITING/TIMED_WAITING 状态线程
thread --state WAITING
thread --state TIMED_WAITING

步骤2：分析阻塞线程堆栈

# 查看阻塞线程详情
thread <blocked-thread-id>

# 查看持有锁的线程
thread -b

步骤3：检测死锁

# 检测 JVM 是否存在死锁
jstack <pid> | grep -A 20 "Found one Java-level deadlock"

# 或使用 arthas 的 thread 命令分析
thread
# 找到状态为 BLOCKED 且长时间不变的线程
thread <id>

步骤4：监控线程创建

# 查看线程创建/销毁趋势
thread -n 5 -i 2000 -c 10

完整示例

# 1. 查看线程状态
thread

# 2. 发现大量 BLOCKED 线程，查看它们持有/等待的锁
thread -b

# 3. 查看具体阻塞线程
thread <blocked-thread-id>

# 4. 如果是死锁，输出 jstack 分析
# 退出 arthas
stop
# 使用 jstack
jstack <pid> > /tmp/jstack.log
grep -A 30 "Found one Java-level deadlock" /tmp/jstack.log

类冲突排查

场景描述

遇到 ClassNotFound、NoSuchMethodError、LinkageError 等类加载相关错误。

排查步骤

步骤1：查找类所在位置

# 查找类被哪个 jar 包加载
sc -d java.lang.String

# 查找特定类
sc -d com.xxx.utils.StringUtils

# 搜索包含特定名字的类
sc *StringUtils*

步骤2：查看类加载器树

# 查看类加载器层次结构
classloader -t

# 查看特定类加载器的信息
classloader -l

步骤3：查看类的详细信息

# 查看类的完整信息（包括 classloader）
sc -d -f com.xxx.utils.StringUtils

步骤4：解决类冲突

# 将冲突的类从某个 jar 中删除或重命名后重新加载
# 使用 redefine 热更新类（见热修复章节）

完整示例

# 1. 报错 NoSuchMethodError: com.google.common.base.Preconditions.checkNotNull
# 2. 查找该类
sc -d com.google.common.base.Preconditions

# 3. 查看类加载器
classloader -t

# 4. 查找所有包含该类的 jar
jad com.google.common.base.Preconditions
# 查看输出中的 class location

方法参数/返回值查看

场景描述

需要在不修改代码、不重启的情况下，查看某个方法的输入参数和返回值。

排查步骤

步骤1：watch 方法调用

# 查看方法参数和返回值
watch com.xxx.service.UserService getUserInfo '{params, returnObj}'

# 查看参数、返回值、耗时
watch com.xxx.service.UserService getUserInfo '{params, returnObj, #cost}'

# 条件过滤：只查看特定参数的调用
watch com.xxx.service.UserService getUserInfo 'params[0].equals("123")' '{params, returnObj}'

# 查看异常
watch com.xxx.service.UserService getUserInfo '{params, returnObj, throwExp}'

步骤2：按指定条件 watch

# 只查看耗时超过 100ms 的调用
watch com.xxx.service.UserService getUserInfo '#cost > 100' '{params, returnObj}'

# 只查看返回值不为空的调用
watch com.xxx.service.UserService getUserInfo 'returnObj != null' '{params, returnObj}'

# 查看异常调用
watch com.xxx.service.UserService getUserInfo 'throwExp != null' '{params, throwExp}'

步骤3：深入查看对象

# 查看对象的详细字段
watch com.xxx.service.UserService getUserInfo '{params[0], returnObj.id, returnObj.name}'

# 调用对象的方法
watch com.xxx.service.UserService getUserInfo 'params[0].toString()'

# 输出为 JSON 格式
watch com.xxx.service.UserService getUserInfo '{params, returnObj}' -x 2

步骤4：限制输出次数

# 只输出前 5 次
watch com.xxx.service.UserService getUserInfo '{params, returnObj}' -n 5

# 输出条件满足的前 10 次
watch com.xxx.service.UserService getUserInfo 'params[0].equals("123")' '{params, returnObj}' -n 10

完整示例

# 1. 监控某个方法的入参出参
watch com.xxx.controller.OrderController getOrders '{params[0], params[1], returnObj}'

# 2. 如果返回值是集合，查看集合大小
watch com.xxx.controller.OrderController getOrders '{params, returnObj.size()}'

# 3. 只查看 userId 为特定值的调用
watch com.xxx.controller.OrderController getOrders 'params[0].equals("user123")' '{params, returnObj}' -x 3

# 4. 查看异常情况
watch com.xxx.service.PaymentService pay '{params, throwExp, #cost}' -e

热修复代码

场景描述

线上紧急 bug 需要立即修复，不能重启服务或等待发版。

排查步骤

步骤1：反编译代码

# 反编译类，查看当前代码
jad com.xxx.service.UserService

# 反编译特定方法
jad com.xxx.service.UserService getUserInfo

# 保存反编译结果到文件
jad com.xxx.service.UserService > /tmp/UserService.java

步骤2：修改代码

# 编辑反编译后的代码
# 注意：需要保持代码结构一致，不能添加新方法/字段

步骤3：编译新代码

# 方式1：使用 jad + mc + redefine 组合

# 1. 反编译
jad --source-only com.xxx.service.UserService > /tmp/UserService.java

# 2. 修改代码后编译
mc -c <classloader-hash> /tmp/UserService.java -d /tmp/

# 3. 查找 classloader hash
sc -d com.xxx.service.UserService
# 记录 classLoaderHash

# 4. 编译
mc -c <classloader-hash> /tmp/UserService.java -d /tmp/

# 5. 热加载编译后的类
redefine /tmp/com/xxx/service/UserService.class

步骤4：验证热修复

# 验证类已更新
jad com.xxx.service.UserService

# 或测试接口调用

注意事项

  • 只能修改方法内部逻辑，不能添加/删除方法或字段
  • 不能修改类签名
  • 需要获取正确的 classloader hash
  • 生产环境谨慎使用，建议先在测试环境验证

完整示例

# 1. 查看需要修复的类
jad com.xxx.utils.DateUtils

# 2. 反编译保存
jad --source-only com.xxx.utils.DateUtils > /tmp/DateUtils.java

# 3. 编辑代码修改 bug
# vim /tmp/DateUtils.java

# 4. 获取 classloader hash
sc -d com.xxx.utils.DateUtils
# 假设输出 classLoaderHash: 12345678

# 5. 编译
mc -c 12345678 /tmp/DateUtils.java -d /tmp/

# 6. 加载新类
redefine /tmp/com/xxx/utils/DateUtils.class

# 7. 验证
jad com.xxx.utils.DateUtils

OOM 排查

场景描述

应用频繁出现 OutOfMemoryError 或内存不足。

排查步骤

步骤1：分析当前内存状态

# 查看内存概况
memory

# 查看堆内存详情
vmtool --action getInstances --limit 10 --className java.lang.Object --expr =@java.lang.management.ManagementFactory@getMemoryMXBean().getHeapMemoryUsage()

步骤2：查找大对象

# 查找数量最多的对象类型
vmtool --action getInstances --className java.lang.String --limit 100

# 查找自定义大对象
vmtool --action getInstances --className com.xxx.entity.Order --limit 20 --displayFieldValues

步骤3：分析对象引用链

# 查找持有特定对象引用的对象
vmtool --action getInstances --limit 1 --className java.lang.ThreadLocal --displayFieldValues

步骤4：dump 堆内存

# 生成堆 dump
heapdump /tmp/heapdump_$(date +%Y%m%d_%H%M%S).hprof

# 如果内存已满，可能需要通过 jmap
# 退出 arthas 后执行
jmap -dump:format=b,file=/tmp/heapdump.hprof <pid>

步骤5：分析 dump 文件

# 离线分析（使用 MAT、jhat 等）
# 或在线查看最可能泄漏的对象
# 查看 Finalizer 队列
jmap -finalizerinfo <pid>

# 查看堆直方图
jmap -histo:live <pid> | head -20

完整示例

# 1. 查看 OOM 前的内存状态
memory

# 2. 查找疑似泄漏的对象
vmtool --action getInstances --className com.xxx.cache.LocalCache --limit 10 --displayFieldValues

# 3. 查看 GC 状态
vmtool --action getInstances --className java.lang.management.MemoryMXBean --limit 1 --displayFieldValues

# 4. 生成堆 dump
heapdump /tmp/oom_dump.hprof

# 5. 退出后用工具分析 dump
# 推荐使用 Eclipse MAT

死循环排查

场景描述

某方法进入死循环，导致 CPU 100%。

排查步骤

步骤1：定位高 CPU 线程

# 查找 CPU 最高的线程
thread -n 5

# 记录线程 ID

步骤2：查看线程堆栈

# 查看线程堆栈
thread <thread-id>

# 如果堆栈显示在特定方法，可能就是死循环位置

步骤3：监控方法调用

# 追踪疑似死循环的方法
trace com.xxx.service.UserService processBatch

# 查看调用次数和耗时
monitor -c 1 com.xxx.service.UserService processBatch

步骤4：方法计时并检测

# 使用 tt 捕获调用，查看是否长时间卡在某个调用
tt -t com.xxx.service.UserService processBatch '#cost > 5000'

步骤5：临时干预

# 如果能热修复，修改代码跳出循环（见热修复章节）
# 或通过修改配置动态关闭相关功能
# 例如：通过 Spring Cloud Config 或配置中心更新配置

完整示例

# 1. CPU 飙升，先看线程
thread -n 10

# 2. 发现线程 5678 在 processBatch 方法中
thread 5678

# 3. 追踪该方法
trace com.xxx.service.UserService processBatch

# 4. 使用 profiler 确认热点
profiler start --event cpu
# 等待 10 秒
profiler stop

# 5. 如果确认死循环，热修复代码跳出循环
jad com.xxx.service.UserService
# ... 修改代码 ...
# redefine 加载新类

SQL 执行慢排查

场景描述

接口响应慢，怀疑是 SQL 查询导致。

排查步骤

步骤1：定位慢查询方法

# 追踪 DAO 方法
trace com.xxx.dao.UserDao findByUsername

# 查找所有 DAO 层方法
trace com.xxx.dao.* *

# 设置耗时阈值
trace com.xxx.dao.* * '#cost > 500'

# 使用 jad 查看 SQL 构建逻辑
jad com.xxx.dao.UserDao

# 或使用 ognl 查看 PreparedStatment 参数
watch com.xxx.dao.UserDao findByUsername 'params'

步骤3：监控数据库调用

# 监控 JDBC 执行
watch java.sql.Connection prepareStatement '{params[0]}'

# 监控执行情况
watch java.sql.Statement execute '{params}'

步骤4：分析连接池状态

# 查看连接池类（如 Druid）
vmtool --action getInstances --className com.alibaba.druid.pool.DruidDataSource --limit 1 --displayFieldValues

# 查看活跃连接数等关键指标

完整示例

trace com.xxx.controller.UserController getUser

trace com.xxx.dao.UserDao findByUsername

# 3. 查看执行的 SQL
watch com.xxx.dao.UserDao findByUsername 'params[0]'

# 4. 查看连接池状态
vmtool --action getInstances --className com.alibaba.druid.pool.DruidDataSource --limit 1 --displayFieldValues

# 5. 如果需要查看实际执行的 SQL，可以：
# 方法1：启用数据库慢查询日志
# 方法2：使用数据库监控工具（如 pmm）
# 方法3：临时修改代码打印 SQL（热修复）

最佳实践

1. 安全第一

  • 生产环境谨慎操作：尽量先在测试环境验证
  • 备份重要数据：执行 heapdump、redefine 前评估影响
  • 记录操作：保留操作日志，便于复盘

2. 高效排查

  • 从宏观到微观：先 dashboard/tread 看全局，再深入具体方法
  • 善用条件过滤：使用 OGNL 表达式减少输出干扰
  • 组合使用命令：trace + watch + stack 配合定位问题

3. 热修复注意事项

  • 不能添加/删除方法或字段
  • 不能修改类签名
  • 需要正确的 classloader hash
  • 热修复仅在当前 JVM 有效，重启后失效
  • 复杂逻辑修改建议走正常发版流程

4. 常用 OGNL 表达式

# 参数/返回值
'{params, returnObj}'
'{params[0], returnObj.id}'

# 条件判断
'#cost > 500'
'returnObj != null'
'params[0].equals("123")'

# 对象方法调用
'params[0].toString()'
'returnObj.size()'

# 异常捕获
'{params, throwExp}'

# 静态方法调用
'@java.lang.System@currentTimeMillis()'

5. 输出控制

# 限制输出深度
watch Service method '{params}' -x 2

# 限制输出次数
watch Service method '{params}' -n 5

# 只看异常
watch Service method '{throwExp}' -e

6. 常见问题处理

问题 解决方案
类加载失败 检查 classloader hash，确保类路径正确
反编译失败 类可能被加密或使用字节码增强
热修复无效 检查类签名是否改变，方法是否新增
输出过多 使用条件过滤、限制次数、调整输出深度
attach 失败 检查用户权限，确认 Java 版本兼容
