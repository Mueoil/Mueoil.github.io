SQL优化 递归查询
发布日期:   2023-11-26

第一种方式用了最为普遍的递归查询，简而易懂，但是查询速度慢（5w条数据查询出来大概10s左右）

好处：

  1. 代码使用了Java 8的流式操作和Lambda表达式，使得代码更加简洁和易读。
  2. 通过使用流式操作和Lambda表达式，可以减少显式的循环和条件判断，提高代码的可读性和可维护性。
  3. 代码将省市区信息转换为了自定义的CnRegionInfo类，使得数据结构更加清晰，方便后续处理和使用。

坏处：

  1. 方法findSubArea是一个递归方法，存在潜在的性能问题。在大数据量的情况下，递归调用可能导致栈溢出或性能下降。
  2. 使用流式操作和Lambda表达式会引入一些额外的性能开销，尤其是在数据量较大时。在性能要求较高的场景下，可能需要考虑其他优化方式。
  3. 代码中对于空指针的处理比较简单，可能存在空指针异常的风险。

private static List<CnRegionInfo> findSubArea(Long pid, List<Ztree> areaList) {

    List<Ztree> subAreaList = areaList.stream().filter(area -> area.getpId().equals(pid)).collect(Collectors.toList());

    return subAreaList.stream().map(item -> {
        CnRegionInfo cnRegionInfo = new CnRegionInfo();
        cnRegionInfo.setCriCode(item.getId().toString());
        cnRegionInfo.setParentId(item.getpId());
        cnRegionInfo.setCriName(item.getName());
        cnRegionInfo.setChildren(findSubArea(item.getId(), areaList));
        return cnRegionInfo;
    }).collect(Collectors.toList());

}

第二种方式将查询的子区域列表放进缓存，但是查询速度也比较慢（5w条数据查询出来大概8s左右）

这个版本的代码在实现加载中国区域省市区信息的功能上与前一个版本类似，不过采用了不同的实现方式。

好处：

  1. 缓存子区域列表：使用subAreaMap缓存了已查询的子区域列表，可以避免重复查询，提高了查询的效率。
  2. 采用传统的for循环和computeIfAbsent方法：与前一个版本使用流式操作和Lambda表达式不同，这个版本采用了传统的循环和方法调用。

坏处：

  1. 编码方式较为传统：相比于前一个版本使用流式操作和Lambda表达式，这个版本的代码显得更为传统，可能不够简洁和易读。
  2. 重复查询子区域列表：在递归查询子区域时，对于每个区域都会通过computeIfAbsent方法查询一次子区域列表，可能存在一定的性能开销。
  3. 方法mapToCnRegionInfo内部的逻辑较为复杂：这个方法负责将Ztree对象转换为CnRegionInfo对象，并递归查询子区域信息，逻辑相对复杂，可读性较差。

/**
 * 递归查询省市区
 * @param ztree
 * @param allZtrees
 * @param subAreaMap
 * @return
 */
private static CnRegionInfo mapToCnRegionInfo(Ztree ztree, List<Ztree> allZtrees, Map<Long, List<Ztree>> subAreaMap) {
    CnRegionInfo cnRegionInfo = new CnRegionInfo();
    cnRegionInfo.setCriCode(ztree.getId().toString());
    cnRegionInfo.setParentId(ztree.getpId());
    cnRegionInfo.setCriName(ztree.getName());

    List<Ztree> subAreaList = subAreaMap.computeIfAbsent(ztree.getId(), id -> allZtrees.stream()
            .filter(a -> a.getpId().equals(id))
            .collect(Collectors.toList()));

    List<CnRegionInfo> subRegionInfoList = new ArrayList<>();
    for (Ztree subArea : subAreaList) {
        CnRegionInfo subRegionInfo = mapToCnRegionInfo(subArea, allZtrees, subAreaMap);
        subRegionInfoList.add(subRegionInfo);
    }

    cnRegionInfo.setChildren(subRegionInfoList);

    return cnRegionInfo;
}。

第三种方式在使用栈实现递归查询方面增加了栈操作，提高了查询效率，降低系统资源消耗，防止可能的栈溢出问题。（5w条数据查询出来大概1s左右）

好处：

  1. 使用栈实现递归查询：这个版本的代码通过使用栈数据结构来模拟递归的过程，避免了真正的递归调用，可以降低系统资源消耗，防止可能的栈溢出问题。
  2. 使用流式操作替代循环和computeIfAbsent方法：与第一个版本相比，这个版本使用了流式操作和groupingBy方法将Ztree列表按照父节点ID进行了分组，并避免了重复查询子区域列表的问题。

坏处：

  1. 栈操作增加了复杂性：相比于前两个版本，这个版本的代码引入了栈操作，逻辑相对复杂，可能会增加代码的理解和维护难度。

 /**
     * 递归查询省市区()
     * @param ztree
     * @param allZtrees
     * @return
     */
    private static CnRegionInfo mapToCnRegionInfo(Ztree ztree, List<Ztree> allZtrees) {
        CnRegionInfo cnRegionInfo = new CnRegionInfo();
        cnRegionInfo.setCriCode(ztree.getId().toString());
        cnRegionInfo.setParentId(ztree.getpId());
        cnRegionInfo.setCriName(ztree.getName());

        Map<Long, List<Ztree>> groupedZtrees = allZtrees.stream()
                .collect(Collectors.groupingBy(Ztree::getpId));

        Stack<CnRegionInfo> stack = new Stack<>();
        stack.push(cnRegionInfo);

        while (!stack.isEmpty()) {
            CnRegionInfo currentCnRegionInfo = stack.pop();
            List<Ztree> subAreaList = groupedZtrees.getOrDefault(Long.parseLong(currentCnRegionInfo.getCriCode()), Collections.emptyList());
            List<CnRegionInfo> subRegionInfoList = new ArrayList<>();
            for (Ztree subArea : subAreaList) {
                CnRegionInfo subRegionInfo = new CnRegionInfo();
                subRegionInfo.setCriCode(subArea.getId().toString());
                subRegionInfo.setParentId(subArea.getpId());
                subRegionInfo.setCriName(subArea.getName());

                stack.push(subRegionInfo);
                subRegionInfoList.add(subRegionInfo);
            }

            currentCnRegionInfo.setChildren(subRegionInfoList);
        }

        return cnRegionInfo;
    }

// 这边注意一个栈的问题
/*
在代码中，cnRegionInfo 对象通过 stack.push(cnRegionInfo); 放入了栈中。然后，通过循环遍历栈中的元素，每次从栈中弹出一个元素并进行处理。

在循环的最后一行代码 currentCnRegionInfo.setChildren(subRegionInfoList);，我们将 subRegionInfoList 设置为 currentCnRegionInfo 对象的子对象列表。

因此，实际上是 currentCnRegionInfo 对象拥有了子对象列表，而不是 cnRegionInfo 对象。由于 currentCnRegionInfo 是从栈中弹出的对象，所以它被赋予了每个层级的子节点信息，并通过 setChildren() 方法来设置。

所以尽管在循环开始前 cnRegionInfo 没有设置 setChildren() 方法，但随着循环的进行，currentCnRegionInfo 被更新并设置了子节点信息，因此最终 cnRegionInfo 对象也将具有正确的子节点信息。
*/

总结与建议

当使用三级联动和父子树递归查询时，对于数据量大的情况，我建议使用版本三的方法，即通过栈模拟递归查询。

理由如下：

  1. 避免栈溢出：由于递归调用可能导致栈溢出的问题，在处理大量数据时，使用栈模拟递归查询可以降低系统资源消耗，避免出现栈溢出的情况。
  2. 减少递归调用开销：递归调用需要频繁地建立和销毁函数栈帧，对于大量数据的处理，递归调用开销可能较大。而栈模拟递归查询通过维护一个栈数据结构，避免了递归调用的开销，可以提高效率。
  3. 流式操作处理子区域的查询：版本三中使用了流式操作和groupingBy方法将子区域列表按照父节点ID进行分组，避免了重复查询的问题。这种方式在数据量大的情况下，可以减少数据库查询次数，提高效率。

需要注意的是，当数据量非常大且层级深度较多时，仍然存在性能问题。在实际应用中，可以考虑优化查询逻辑，比如引入缓存机制、使用分页查询等，以满足大规模数据查询的需求。
